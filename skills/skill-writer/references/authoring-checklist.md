# skill-writer authoring checklist

Use this checklist when writing a new Codex skill or heavily rewriting one.

## 1. Scope + boundary preflight

Answer these before drafting prose:

- What exact user intent should trigger this skill?
- Is this a **new intent surface**, or should an incumbent skill be extended?
- Is the skill an **owner**, a **gate**, or an **overlay**?
- Which adjacent skills are most likely to be confused with it?
- Does it need session-start wording?

Prefer **incumbent-first** unless one of these is true:
- owner/gate/overlay role changes
- runtime assumptions differ materially
- a new sibling is required to keep discovery precise

## 2. Description rubric

### First-line brief
- Target: **≤ 120 chars**
- State what the skill does, not its whole process
- Use concrete nouns and verbs

### Full description
- Recommended: **180–450 chars**
- Include **what** the skill does and **when** it should be used
- Include real user phrases, file types, tools, or artifacts
- Include one boundary hint when confusion is likely
- Avoid synonym stuffing and repeated examples

Weak:
- "Helps with skills."
- "Useful for documentation."

Stronger:
- "Write new Codex skills with concise descriptions, sharp boundaries, token-aware packaging, and framework embedding."

## 3. Body checklist

Minimum sections:
- `## When to use`
- `## Do not use`
- `## Task ownership and boundaries`
- `## Required workflow`

Add only what materially improves execution. Good optional sections:
- `## Output defaults`
- `## Hard constraints`
- `## Trigger examples`
- `## Reference map`

## 4. Progressive disclosure + token control

Keep the cheapest useful layer at the top:
1. `description` for routing
2. `SKILL.md` core workflow for execution
3. `references/` for detail that is not always needed
4. `scripts/` or `assets/` only when deterministic reuse matters

Guidelines:
- Keep `SKILL.md` compact; move long matrices and examples out
- Avoid nested references beyond one level
- Prefer scripts when the same deterministic logic would otherwise be regenerated repeatedly
- Keep startup-visible metadata dense and cheap

General external guidance from Agent Skills docs also supports keeping the main
skill file relatively short and moving heavier detail out of the startup path.

## 5. Framework embedding checklist

Before treating the skill as done, confirm:
- `routing_layer`
- `routing_owner`
- `routing_gate`
- `session_start`
- folder slug matches `name`
- routing index or layer docs are updated if the new skill changes discovery
- sync and validation commands are queued
- maintenance-trigger phrases are present when the skill owns maintenance or framework entry behavior

## 6. Trigger-eval checklist

Prepare both kinds of prompts:

### Should trigger
Create 5–10 prompts that should clearly match the skill.

### Should not trigger
Create 5–10 prompts that belong to adjacent skills.

For each false positive or false negative, tighten one of:
- description wording
- `## When to use`
- `## Do not use`
- boundary language with neighboring skills

## 7. Repo validation commands

Run after edits:

```bash
cargo run --manifest-path scripts/skill-compiler-rs/Cargo.toml -- \
  --skills-root skills \
  --source-manifest skills/SKILL_SOURCE_MANIFEST.json \
  --health-manifest skills/SKILL_HEALTH_MANIFEST.json \
  --apply
```
