# Claim–evidence decision ladder（主张—证据决策阶梯）

跨 **`$paper-workbench` / `$paper-reviewer` / `$paper-reviser`** 的真源：**默认反对用「降 claim / 缩口径」代替「该补的证据与分析」**。叙事上按**强对抗审稿**校准：先问「最坏审稿人会怎么卡」，再写建议。缩主张是**低成本改稿**，但在许多场景下是对科学贡献的**逃避**；技能栈必须把顺序写死，避免模型图省事。

## 第一性原理

- 审稿人否决的多数是 **证据不足、比较不公平、可复现性不够**，不是「英语不够华丽」。
- **语言与结构上降 claim** 往往比跑一个对照实验便宜几个数量级；若不做阶梯判别，系统会系统性偏向偷懒解。
- **诚实**不等于「尽快变谦虚」；诚实等于 **在可行范围内先把证据顶到与主张匹配**，再承认剩余局限。

## 默认顺序（从高到低尝试）

在给出以 **缩口径 / 降 claim / 改投更低栏** 为主轴的建议之前，必须先穷尽（或显式否决）更靠前的台阶：

1. **补证据与补强分析（最小可发表增量）**
   缺基线、缺对照、统计不稳、消融不完整、失败案例缺失、敏感性/外部校验、代表性样本、泄漏与可复现披露等。输出须能写成 **可执行清单**：每条对应「补什么 → 预期支撑哪条 claim → 预期落在哪张图/表」；若可判断，附**工作量级**（小/中/大），禁止用「太难」一笔带过。

2. **强化论证与比较公平性（不必然新实验）**
   更公平的协议描述、更直接的图表、更清晰的 threat to validity 回应、理论稿的定理/证明闭合。能**用现数据重分析或重画图**关掉的洞，优先于改口。

3. **呈现与结构（不削弱已支持结论）**
   删冗余、压篇幅、附录化细节、改善叙事节奏——**不得**用 limitation 段落替代本该做的实验。

4. **缩 claim / 降 ceiling / appendix 弱化主文 / 换 venue**
   仅当 §「何时允许以缩口径为主」成立，或用户**显式**选择「不补实验、只改主张/换栏」时，才把本层作为**主**建议。

## 禁止的「降 claim 偷懒」模式（默认须抓）

- **无证据优先选项**：`evidence gap` 仍是 B 类（需补数据/分析/基线）时，主结论却是「把主张改弱就好」。
- **把难度当不可能**：「要算力 / 要时间」但未拆最小增量、未说明哪一步可砍掉或降级为附录实验。
- **Limitation 代替实验**：用一段 limitation 吞掉本该做的对照或基线。
- **沉默降档**：改了 abstract/intro 的 claim level，却无对应的 `claim_ledger_delta` 与**阶梯理由**（见下）。
- **代码质疑靠嘴**：审稿人要复现 / 对齐实现 / 查 bug / 核验复杂度——回应却是「将实现公开」「upon request」「团队已核验」却无 **commit/tag、环境锁、最小复现命令、或正文—代码逐点核对表**，视同逃避。
- **数学质疑靠软改**：审稿人指出缺引理、错误不等式、错误条件——只把定理改成一段话、只说「我们更谨慎表述」却不给 **补丁证明 / 更正条件 / 反例收窄 / Conjecture 显式化**，视同逃避。

## 何时允许以「缩口径 / 放弃原主张」为主轴

必须能指向至少一类**实质性阻塞**（并在输出里写明 `narrowing_is_primary_because:`）：

- 主张在逻辑或领域常识上**不可辩护**，且团队接受收缩。
- **伦理、法律、数据不可得、设备不可达**等硬阻塞，短期无法缓解。
- **核心假设被否**或关键效应为负，但仍可诚实地报告收缩后的信息。
- 用户**明确**声明资源/时间/合作上限，要求「在不补实验前提下改到可投」。
- 已列出证据优先路径且用户**明确拒绝**执行（记录为「用户选择 narrow」而非「模型代选」）。

## `$paper-reviewer` 输出硬契约

每当存在 **B 类** blocker 或顶刊/顶会栏位要求时，在 `next_honest_move` **之前或并列**给出：

```text
evidence_first_options:
  - <minimal add-on 1 + which claim it closes + rough cost>
  - <minimal add-on 2 + ...>
narrowing_is_primary_because: <null | 指向 §「何时允许」之一>
```

- 若 `narrowing_is_primary_because` 非空，**禁止**省略 `evidence_first_options`：仍须简述「考虑过哪些增量、为何仍不足以顶住栏位」，避免假二分。
- 对用户可见表述保持审稿语言；**不要把本文件名或「阶梯」元话术**塞进主答案，除非用户要协议工件。

## `$paper-reviser` 执行硬契约

- Findings 仍判定为 **「可用补充证据/分析闭合」** 时：**主批次默认 repair**（按意见补图、补表、补方法、补统计表述），**不得**把整轮改成单纯降 claim 了事。
- **narrow / delete / de-emphasize** 可与 repair 同轮，但须有用户或 reviewer 侧的 **`narrowing_is_primary_because`**（或等效记录），否则 narrow 只能作为**备选补丁**列在「若不做实验则…」。
- Rebuttal 场景：若审稿人点名要实验/对照，**优先给可执行补做路径**；仅当不可行时再用 limitation + 缩 claim，并写明不可行理由。

## 审稿意见 / R&R：逐条关停与逃逸红线（硬性）

凡是任务中出现 **reviews、meta-review、AC 评语、程序化 Major/Minor checklist**（含用户粘贴回信模板）的整体或混合改稿：**以下默认强制**，不得用「措辞更谦虚」「多写 LIMITATION」作为主要交付。

1. **逐条绑定**：每个 `point_id` 至少映射到以下之一：**(a)** 手稿/图/表/附录/方法段落上的**可定位修改**；**(b)** 已完成的分析与实验增量及其在主文或补充材料中的**落点**；**(c)** `cannot_fix_because`（硬阻塞，如伦理、数据不可得、设备不可达），并给出可审计的一句话理由链。
2. **默认可修优先**：意见要求补基线/消融、澄清协议/SWE、补强统计与不确定性、放出实现与数据索引、重做图表或公允比较 → **归为 repair 主轴**；**禁止**把主回应写成「我们承认局限并下调语气」却不对证据结构动手。
3. **禁止 rebuttal-only**：response letter 若声称「已完全回应」，须能对照到 **可 diff 的手稿或补充材料变更**；不得仅凭辩论文本过关。
4. **语义逃避（视同违反本阶梯）**：包括：删掉最高比较级形容词但实证覆盖未增加；把本应主文呈现的 decisive comparison **藏附录**却仍保留同等贡献叙事；用更长 caveat **顶替**被要求的新对照或测量；在多轮回合中 **silent drift** abstract/intro 而未登记 `claim_ledger_delta`。

## 代码/实现质疑（高逃逸风险硬闸）

审稿意见涉及 **可复现性、源码与正文/伪代码一致性、默认超参、随机性种子、复杂度或资源声明、潜在实现 bug** 等时，默认 **repair**，且关停件须**可被第三方复核**，不得仅靠叙事。

- **必须提供或可执行其一**：**(a)** 版本化工件（repo **commit/tag**、环境锁 `conda.lock`/`requirements.txt`/Docker、`README` **最小命令**跑出论文主表一行或主图一案）；**(b)** 与正文算法框 **逐步对照表**（伪代码行 ↔ 源码符号/文件名/函数）；**(c)** 若不能完整开源：`cannot_fix_because` **极严格**（如法律禁令）+ **仍须**给的 **minimal reproduction**（裁剪数据 + 单机脚本 / synthetic fixture）—不得空口。
- **禁止**：仅以「已向编辑部承诺稍后公开」「审稿结束即上传」「作者将邮件发送代码」作为主要回应而无**立即可核验**锚点。
- **发现不一致**：若在核对中发现实现 ≠ 手稿叙述，优先 **更正正文方法/复杂度/默认值**并披露对结果影响的范围；不得用含糊 limitation 糊弄。
- **rebuttal 写法**：点对点须含 **可查链接或附件清单**（Zenodo/GitHub/tag + hash）或等价 bundle，不得整段泛泛「增强可复现性」而无具体物。

## 数学/推导质疑（高逃逸风险硬闸）

审稿意见涉及 **证明缺失、推导错误、条件过强/过弱、交换极限合法性、常量与阶记号、可分性可测性、不变量或等价表述**等时：**证明与符号是主战场**，不是润色。

- **必须交付其一**：**(a)** 完整补丁证明（含新增引理/附录重排，标出改动的 `eq.`/`Thm.`）；**(b)** **勘误式**修订（定理陈述收紧/结论减弱 + 全篇依赖该结论的同步修改 + `claim_ledger_delta`）；**(c)** 审稿人正确时给出 **反例或显式反证边** 并相应 **收窄可主张范围**；**(d)** 无法修复时 **显式降级**为 conjecture / open problem，并删除正文中暗示已证的措辞。
- **禁止**：用「更非正式」「直觉上成立」「显然」「standard argument」替代缺步；用「notation 统一」掩盖 **实质错误**；把错误定理 **无声改成** Discussion 里一句话而不说明 **推理链已变**。
- **与代码交叉**：若质疑来自「公式与实现不等价」，须同时走 **§代码/实现质疑** 的核对路径，两边一起关停。

## 与顶刊栏的关系

目标栏越高，**证据优先选项的默认权重越高**：顶刊不是「写谦虚一点」能换的；**先补 decisive evidence**，再谈叙事与语气。

## 与用户显式指令

用户说「不要补实验，只降 claim」→ 尊重，但须在内部记录为**用户选择的 narrow 路径**，避免与「模型偷懒」混淆。

用户说「顶住主张、该补就列出来」→ 本文件为**强制默认**。
