# 科研写作用语长期规范（跨 lane 真源）

本文档是 **`$paper-workbench` / `$paper-writing` / `$paper-reviser` / `$paper-reviewer`** 共用的**长期运行**约束：降低「术语堆砌、自造概念、同义反复」导致的拗口稿与不稳定「精准修改」。多轮会话中默认遵守，除非用户显式豁免。

## 目标

- **可读性**：读者能把它当成该子领域的正常论文读下去，而不是「模型腔」或内部黑话。
- **可核验**：重要用语能在标准教材、方法论文或目标子领域高频文献里找到对应，而不是未定义的组合词。
- **可稳定改稿**：`surgical` 轮次只动允许范围内的字句；术语表冻结后，后续修改不偷偷换名或换概念。

## 1. 用语对齐（禁止「听起来专业」的生造）

- **优先子领域通行词**：先选定 2–3 篇「对标论文」（同目标期刊 / 同类问题 / 相近方法），**名词、动词框架、常见缩写**尽量与之一致；需要偏门说法时，说明为何与对标文献不同。
- **禁止未锚定的复合指标名**：形如「X–Y ratio」「A–B–C score」且**文献中无稳定同名**时，不得当常识反复使用。处理方式（择一）：
  - 换成该领域**已有指标名**并引用出处；或
  - 在方法段**一次**给出可操作定义 + 记号（符号/下标），正文后文只用该记号或一句人话指代。
- **多义词必须落语境**：如 *retention*、*saturation*、*coverage*、*capacity* 等，在首次引入时钉死含义（例如：用户留存 vs 权重记忆 vs 缓存驻留），避免跨段漂移。
- **中英混排纪律**：中文稿里的英文术语首次出现可括注英文；英文稿避免无必要的中式直译硬造名词。

## 2. 术语密度与重复（反车轱辘话）

- **每个非平凡概念**：全文只做**一次**「正式引入」（定义或引用 + 记号）。之后用缩写、符号或短指代；不要在 Results / Discussion 里用不同比喻把同一概念再讲三遍。
- **段落级**：相邻段落不得重复同一论证链（同一「问题—手段—收益」模板复制粘贴）；若必须呼应前文，用**一句**回溯 + 新信息增量。
- **章节级**：abstract、introduction、discussion 可以呼应，但**禁止同句同 claim 的改头换面重复**；讨论局限与贡献时各写各的，不共享同一长句骨架。

## 3. 姿态与语气：禁止内部口径、防御口径、堆叠负面对比

论文默认写给**领域读者**，不是写给内部项目组做免责，也不是预演审稿辩论。

- **内部口径**：避免像在同步站会一样的句子（未解释的流程代号、只有团队懂的动机、过程细节压过科学问题）。该删则删，或改写成读者可理解的「问题—做法—结果」。尤其禁止：
  - **把代码命名搬进正文**：函数名、变量名、模块/类名、仓库内路径、临时脚本名、内部缩写（除非已是领域公认记号）。应改为**符号、算法名（若已是文献通用名）、或一句人话描述**；必要时在方法或附录给出实现对照表，而不是在 Results 里出现 `compute_loss_v2` 式标识。
  - **用工程产物当引用**：如「完整结果见 `foo.csv`」「详见 `runs/exp37/output.csv`」——读者无法从论文本身读结论。应改为正文中的**表 / 图 / 摘要统计句**，补充材料用「Table S* / Fig. S* / Supplementary Data」等期刊约定结构；若必须给原始表，放在**正式补充材料**并编号，而不是裸 `.csv` 文件名指向本地路径。
- **防御口径**：避免为防误读而**层层设防**（同一观点用多种否定句式重复兜底），读起来像在辩诉。边界与局限**各用一句**说清楚即可，不叠「律师函」式从句。
- **负面口径 / 对比成瘾**：避免以**否定与对照**为主骨架铺陈贡献（整段由「不是…」「并非…」「我们不…」「…, but …」「rather than …」串起来）。读者应首先看到**你做了什么、证据是什么**，再按需最小化对比。

### 对 `but` / `not` / `rather than` 的纪律

- **默认优先**：正面主句陈述贡献或结果；范围与假设用短句限定（*under …; we assume …; we focus on …*），不必先否定一整类工作再落笔。
- **限制堆叠**：连续句子里不要形成「否定链」（*not A, not B, but C* 重复多轮）。同一段落内，**至多保留一处**强对比转折；其余合并或移到 Related work / Limitations 各写一条。
- **`rather than`**：仅用于**必要**的方法分叉说明（读者会真混淆时），禁止当句式填充；能用「我们采用 X 因为 …」说清的，不要用「我们不用 Y」起头。
- **合法例外**（仍须克制篇幅）：Related work 中的精确区分、Limitations 的诚实边界、与某一强 baseline 的必要对照——用**可核验**的一句话对比，不展开成防御性散文。

### 修稿与审稿回应：不得以「话术防御」顶替实质修改（硬）

本节与上文「减少辩护腔」同一立场：**limitation/caveat 只承担诚实边界**，不得当 **逃避审稿要件** 的替身。

- **禁止**靠加长免责从句、堆叠假定范围、反复强调「我们不声称 X」「结果可能受 Y 影响」等，来**消解**审稿人明示可做的对照、消融、协议澄清或可复现披露；该补证据与分析走 **`claim-evidence-ladder`**，先改证据结构再收束语气。
- **禁止「口头降claim」**：仅把结论写得更谦逊或更弱，却让图表与实验设计仍支撑旧叙事；凡动 claim ceiling，须有 **`claim_ledger` / evidence anchor** 闭环，不得语气单独漂移。
- **回复信（point-by-point）**：承认局限、感谢意见 ≠ 本条已关；须在文字中指向**手稿中的具体改动位点**（节、图号、附录编）或可执行 **补证计划**（何事、落点何处），避免空泛认同。
- **代码质疑**：不得在回复或正文中用大段「增强透明度的承诺语言」顶替 **tag/hash、最小命令复现、或算法—实现对照表**（见 [`claim-evidence-ladder.md`](claim-evidence-ladder.md) §代码/实现质疑）。
- **数学质疑**：不得在不敢补证时用「我们更谨慎措辞」糊弄；须指向 **补丁证明、定理勘误、或显式 conjecture 降级** 与相应正文同步（见同文件 §数学/推导质疑）。

`$paper-writing` 润色时：`surgical` 下遇到上述堆叠，优先**改写为正面主句 + 一条边界**；`refactor` 下可重组段落骨架，但仍不得用更「响」的否定句变相抬 claim。

## 4. 与 `claim_ledger` / 证据锚点配合

- 若某用语绑定特定 claim（例如某指标支撑某结论），在 `claim_ledger` 或等价记录里写明 **allowed_claim_level** 与 **required_evidence_anchor**；后续改稿**不得**通过换更「响亮」的词提升 claim。
- 引入新指标名 = 可能的新科学主张：先走 **`$paper-reviewer` / 主决策链** 是否允许，再走 `$paper-writing` 落笔。

## 5. 「精准修改」（`edit_scope: surgical`）下的用语稳定性

- **默认不改术语体系**：surgical 轮次只做用户锁定范围内的 clarity / flow；若要批量替换某一术语，用户须在 `scope_items` 中写明「术语替换：A→B」或升级为 `refactor`。
- **禁止「全篇统一别名」**：不得借 readability 之名在同一轮对**未列入** `scope_items` 的段落做同义词替换、英美拼写统一、引号体系统一；那是 **静默全局替换**，见 [`edit-scope-gate.md`](edit-scope-gate.md)。
- **禁止整段重贴当交付**：用户只点名一小段却收到「整节润色版」→ 违反门控默认交付形态；应改为带 `change_id` 的局部块或 patch。
- **推荐可选工件**（多轮写作时由 front door 或用户维护）：
  - `paper_story/TERMINOLOGY_GLOSSARY.md`：**术语 / 符号 / 禁止混用别名 / 首次出现章节**。
- 任一 surgical 输出若触及术语一致性，在交付末尾用**一条** `terminology_note` 说明动了哪些词、是否已对齐 glossary（无需长文）。

## 6. 审稿维度（`$paper-reviewer`）

在「语言 / 可读性」或整篇严审中，把本节当作**硬检查项**（优先级在 claim 安全之后、纯辞藻之前）：

- 是否存在**未定义或文献不支撑**的复合命名？
- 同一观点是否在多段**重复表述**而无新信息？
- 高频词是否**语境漂移**（同一英文词前后指代不同对象）？
- 是否出现**防御式堆叠**：密集 `but` / `not` / `rather than`、整段以否定对照撑结构、或内部站会式表述？
- 正文是否出现**未消化的代码标识符**或**裸数据文件名/路径**（如直接「见某 `.csv`」）替代应有叙述与表格？

发现问题时，按严重程度归入现有 A/B/C 类，但结论须**可操作**：删重复、换标准术语、合并定义、或升格为方法段一次定义。

## 7. 与用户豁免

用户明确说「允许新造指标名 / 允许营销式命名 / 不用对齐文献 / 允许强防御语气」时，可在该任务内豁免 §1–§2 或 §3，但须在交付中注明**豁免范围**，避免多轮 silent drift。
