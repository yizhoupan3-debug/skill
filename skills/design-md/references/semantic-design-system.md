# Semantic Design System Notes

`DESIGN.md` 的作用不是做一份“审美点评”，而是把真实界面的设计语言翻译成后续生成可以持续复用的规则文件。

## 核心原则

1. 先看整体气质，再看局部 token。
2. 先抽语义，再补精确值。
3. 颜色、字重、圆角、阴影都要说明“它在系统里负责什么”。
4. 技术值需要翻译成自然语言，避免只留下 `rounded-xl`、`shadow-sm` 这种对后续生成帮助很弱的字面标签。
5. 最终文档必须能支撑后续页面继续生成，而不是只解释当前页面。

## 推荐抽取顺序

1. 产品身份与页面类型
2. 氛围与密度
3. 主次色及功能角色
4. 字体与层级
5. 几何语言
6. 深度与材质
7. 组件签名
8. 布局原则
9. 生成护栏

## 常见失败方式

- 只列 CSS 值，不解释设计含义
- 只写抽象形容词，不给关键精确值
- 把“当前这个页面怎么长这样”误写成“整个系统都应该这样”
- 观察很多，但没有整理成可复用规则
- 没把后续实现和生成阶段需要遵守的禁区写清楚
